DB Pooling and Listening

bb8-pooled short-lived connections, plus one shared, reconnecting listener connection per backend

Problem

Both Postgres and Redis need two different shapes of connection: short-lived checkouts for regular commands/queries, issued concurrently from many call sites, and one long-lived connection for push notifications (Postgres LISTEN/NOTIFY, Redis pub/sub), watched for as long as any channel is subscribed. A bb8 pool is built for the first shape and wrong for the second — a pooled connection can be recycled or reused for an unrelated command mid-subscription. Each backend's connection manager owns both: a bb8 pool of command connections, and a single shared listener that every pooled connection can reach to add or remove subscriptions.

Both listeners share this shape: a keep-alive task waits on a Notify, connects, and spawns a streaming task that reads notifications until the connection drops — at which point it notifies the keep-alive task to try again. Because every channel shares one connection, a drop pauses all subscriptions at once until reconnect completes and every channel is re-subscribed; this is the cost of sharing one socket, traded for not needing a socket per subscribed channel (Postgres) or per listen() call (Redis, before this pattern).

Key rules

QuestionAnswer
When does a command connection get created? Lazily, on the first pool checkout that needs one, up to the pool's max_size. Idle ones are reused across checkouts.
How many listener connections exist? At most one per connection manager — shared by clone across every pooled connection — regardless of how many channels are subscribed.
Can a channel be unsubscribed? Yes — unlisten(channel) removes the handler and, if connected, issues the backend's unsubscribe on the shared connection.
What happens if the shared connection drops? The streaming task ends, which notifies the keep-alive task; it reconnects and re-subscribes every channel still in the handler map. All subscriptions pause together during the gap.
Can subscriptions change while messages are being read? Yes — reading and subscribing use independent halves of the connection (Postgres: a raw command connection alongside the notification stream; Redis: PubSub::split() into a Sink and a Stream), so listen()/unlisten() don't block dispatch.
Regular commands on a pooled connection Go through the backend's normal connection type (Postgres: tokio_postgres::Client/Transaction via Deref; Redis: MultiplexedConnection via ConnectionLike/Deref/DerefMut), entirely separate from the listener connection.

Same shape, different primitives underneath

PGListener and RedisListener independently converged on the same internal shape: a lock-guarded state struct holding Option<write-half> + HashMap<String, BoxedHandler>, a keep-alive task looping on a shared Arc<(Notify, AtomicBool)> that connects and re-subscribes every registered channel, and a streaming task (spawned fresh per successful connect) that reads notifications, dispatches by channel name, and notifies keep-alive on exit.

What differs is backend plumbing, not the pattern: Postgres opens a raw tokio_postgres socket connection and issues LISTEN "channel"/ UNLISTEN "channel" as SQL commands on it directly — there's no separate read/write split, since tokio_postgres::Client already lets commands and the notification stream run concurrently. Redis instead calls Client::get_async_pubsub() then PubSub::split() to get an explicit PubSubSink (subscribe/unsubscribe) and PubSubStream (message reads), because the underlying pub/sub connection type doesn't expose that concurrency for free the way tokio_postgres does. Dispatch key is Notification::channel() (Postgres) vs. Msg::get_channel_name() (Redis); payload is the raw notification string (Postgres) vs. Msg::get_payload::<String>() (Redis).
